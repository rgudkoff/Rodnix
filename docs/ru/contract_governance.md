# Contract Governance (Unix Layer)

Статус: active  
Дата: 2026-03-08

Цель: зафиксировать правила, по которым контрактные тесты (CT-xxx)
гейтят изменения в `kernel/unix` и связанных POSIX entrypoints.

## 1. Merge policy

Для merge в основную ветку обязательны:

- все `CORE`-контракты в статусе `AUTO` должны быть `PASS`;
- итоговый маркер contract-режима: `[CT] ALL PASS`;
- не допускается деградация `AUTO -> MANUAL/PLANNED` без ADR и апдейта модели.

Допустимы к merge:

- `PLANNED` контракты без реализации теста (как технический долг);
- внутренний рефакторинг, не меняющий наблюдаемую CT-семантику.

## 2. Когда нужен новый CT-идентификатор

Новый `CT-xxx` обязателен, если изменение:

- добавляет новый внешний инвариант поведения `spawn/exec/wait/exit/fd/path`;
- меняет contract surface (`unix_layer.h`, userspace ABI ожидания);
- добавляет новую обязательную ветку отказа/безопасности на syscall boundary.

Если меняется только внутренняя реализация без изменения наблюдаемого
контракта, новый `CT-xxx` не требуется.

## 3. Обновление модели и матрицы

Любое изменение в:

- `docs/ru/unix_process_model.md`
- `docs/ru/unix_layer_contract.md`

должно сопровождаться обновлением:

- `docs/ru/unix_process_contract_tests.md` (матрица + статусы);
- тестовой реализации (`userland/*`, `scripts/ci/contract_qemu.sh`) при
  изменении `AUTO`-контрактов.

## 4. Классы контрактов

- `CORE` — обязательный CI-gate.
- `LIFECYCLE` — модель состояний и reap/wait семантика.
- `EXEC` — `spawn/exec` invariants.
- `FD` — дескрипторы и их валидность.
- `PLANNED` — зарезервировано, не блокирует merge.
