# Детерминизм и управляемость отказов

Этот документ описывает минимальную модель ошибок и аварийных остановок
в RodNIX. Цель — обеспечить предсказуемое поведение системы: либо работает,
либо падает **понятно и воспроизводимо**.

## Принципы

- Ошибки не маскируются: fail‑fast лучше скрытой порчи.
- Каждая критическая ошибка фиксируется в логах и завершается panic.
- Инварианты считаются “жёсткими контрактами”.
- В debug‑сборках нарушения инвариантов должны приводить к немедленному падению.

## Модель ошибок

- Успех: `RDNX_OK` (`0`).
- Ошибки: отрицательные значения из `include/error.h`.
- Новые подсистемы обязаны использовать **единый набор кодов** (не свои локальные).

## Fail‑fast инструменты

В коде доступны:

- `PANIC(fmt, ...)` — аварийная остановка с контекстом `file:line`.
- `BUG_ON(cond)` — если условие истинно, система падает.
- `PANIC_IF(cond, fmt, ...)` — условная паника с сообщением.
- `ASSERT(...)` — ассерты уровня debug (см. `include/debug.h`).

## Инварианты (пример)

Ниже — типовые инварианты, которые должны быть зафиксированы по подсистемам:

- **Boot/Memory:** physmap готов до отключения identity map.
- **Scheduler:** current_thread не NULL при переключении.
- **IPC:** длина сообщения валидна и не превышает лимит.
- **VFS:** путь не NULL, root всегда существует.

## Требования к диагностике

Минимальный набор при panic:

- сообщение об ошибке;
- контекст `file:line`;
- консольный вывод (serial + VGA, если доступны).

Дальнейшее расширение (план):

- дамп регистров;
- backtrace;
- сохранение трассировки последних событий.
