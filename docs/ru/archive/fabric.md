# Fabric: шины, устройства, драйверы, сервисы

## Цели

- Централизованный, прозрачный путь для подключения устройств.
- Изоляция драйверов от специфики обнаружения.
- Публикация сервисов (например, клавиатуры) для потребителей.

## Базовая модель

- Bus публикует устройства.
- Driver сопоставляется с устройством.
- Driver может публиковать Service.
- Consumer запрашивает Service по типу.

## Границы ответственности (жёсткие)

- **Bus** отвечает только за обнаружение и публикацию устройств.
- **Device** — структурированное описание устройства, без бизнес‑логики.
- **Driver** отвечает за инициализацию устройства и экспонирует сервисы.
- **Service** — стабильный интерфейс для потребителей (ops + context).
- **Consumer** работает только через сервис, не обращается к железу напрямую.

## Что уже есть

- Реализован базовый Fabric слой.
- Fabric инициализируется до запуска shell.
- Есть виртуальная шина, PCI и PS/2 публикация.
- HID клавиатурный драйвер публикует сервис "keyboard".

## Что планируется

- Расширение списка устройств и драйверов.
- Сервисный слой как основной путь доступа к устройствам.
- Удаление прямых обращений к устройствам вне Fabric.
- Явная версия интерфейса сервиса: `*_ops_v1` и проверка размера struct.
- Service lookup возвращает `ops + context` без универсальных “variant” слоёв.

## Инварианты

- Устройство должно появляться через конкретную шину.
- Драйвер не должен напрямую искать устройства вне Fabric.
- Все сервисы должны быть объявлены и типизированы.
- Каждый сервис содержит ABI‑заголовок (`abi_version`, `size`).
- Ops‑структуры сервисов также содержат ABI‑заголовок.

## Контракты

- Публичные интерфейсы Fabric должны жить в `include`.
- Драйверы обязаны регистрироваться через API Fabric.

## Где смотреть в коде

- `kernel/fabric` и `drivers`.
