# Модель памяти RodNIX

Этот документ описывает формальную модель памяти для RodNIX:
владение, жизненный цикл, неизменяемые области и поведение при OOM.

## Владение памятью

- **PMM** владеет всей физической памятью и является единственным источником
  истины о том, какие страницы свободны/заняты.
- **VM map** управляет виртуальными диапазонами адресов процесса.
- **VM object** управляет страницами диапазона и их backing-store.
- **pmap** управляет аппаратными таблицами страниц (MMU/TLB).
- **Heap/allocators** работают поверх VMM и не минуют PMM напрямую.

Инвариант разделения:
- Политика (map/object) отделена от механизма (pmap).

## Жизненный цикл

1. Boot → ранний physmap.
2. PMM инициализирован по карте памяти.
3. Инициализация VM map/object/pmap для kernel space.
4. Инициализация user VM для первой задачи.
5. Освобождение init‑секций после старта.

## Неизменяемые области

- Таблицы страниц, задействованные в long mode переходе.
- Ядро (текст/rodata) до явного отключения/перемаппинга.
- Initrd до завершения импорта в RAMFS.

## OOM (Out‑Of‑Memory)

Текущая политика:

- **Критические** выделения (heap init/grow) вызывают `panic` (fail‑fast).
- **Некритические** выделения могут возвращать `NULL`, вызывающая сторона обязана
  обработать ошибку.
- OOM фиксируется в trace‑буфере (последние события).

Планируется:

- Централизованный OOM‑обработчик.
- Метрики по источникам OOM и деградационным режимам.
- Разделение OOM по слоям:
  - PMM exhausted,
  - VM object growth failed,
  - pager/backing-store unavailable.

## Модель page fault (целевой контракт)

1. `fault(addr, access, mode)` проверяет принадлежность адреса `vm_map`.
2. Если запись в COW-страницу, выполняется private copy и remap.
3. Если страницы нет в resident set объекта, объект запрашивает fill.
4. При успехе выполняется `pmap_enter`.
5. При отказе:
   - user mode: возвращается ошибка/сигнал,
   - kernel mode: fail-fast (`panic`) для невосстановимых путей.

## Состояния страницы (минимальная модель)

- `FREE` -> `ALLOCATED` -> `ACTIVE` -> `INACTIVE` -> `FREE`
- Дополнительно:
  - `WIRED` (не подлежит вытеснению),
  - `COW_SHARED` / `COW_PRIVATE`,
  - `PAGER_BACKED` (страница может быть восстановлена из backing-store).

Переходы должны быть журналируемы в debug-режиме.

## Порядок блокировок (план)

Чтобы избежать дедлоков, фиксируется порядок:
`vm_map lock -> vm_object lock -> pmap lock`.

Обратный порядок запрещён.

## Освобождение init‑секций

Init‑секции (`__init_start`/`__init_end`) освобождаются после завершения
инициализации ядра. Это освобождает память, не нужную после старта.
