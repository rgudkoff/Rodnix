# Инженерные соглашения

## Общие принципы

- Предпочитаем ясность кода и простые конструкции.
- Избегаем избыточных абстракций без реальной выгоды.
- Любые архитектурные изменения фиксируем в документации.

## Структура и границы

- Arch-независимый код не должен тянуть arch-специфичные заголовки.
- Публичные интерфейсы подсистем — в `include`.
- Внутренние реализации — в соответствующих `kernel/*` каталогах.
- Файловые системы, сетевой стек и драйверы реализуются в ядре
  (не в user space).
- Подсистемы связываются только через явные интерфейсы.
- Запрещены неявные зависимости между подсистемами (\"знать о внутренностях\").
- Для межслойных API используется IDL и автогенерация стаба/диспетчера.

## Ошибки и диагностика

- Используем единые механизмы логирования и паники.
- Ошибки должны быть воспроизводимы и иметь контекст.
- Диагностика не должна ломать критические пути (IRQ, early boot).
- Нарушение контрактов подсистем должно быть диагностируемо.
- Проверки параметров на границах подсистем — обязательны.

## Рефакторинг

- Любые большие перемещения кода сопровождаем краткой записью в
  `development_plan.md` или отдельным заметкой в `docs/`.
- Не возвращаем параллельные подсистемы, уже удаленные.

## Проверки

- При изменениях в подсистемах, которые ранее дублировались, запускаем
  скрипт `scripts/check_parallel_subsystems.sh`.
