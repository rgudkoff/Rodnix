# Архитектура

## Общая модель

RodNIX строится как 64-битное ядро с четким разделением на
архитектурно-независимую часть и архитектурно-специфичные реализации.
Поверх базовых подсистем постепенно выстраивается слой Fabric, который
управляет шинами, устройствами, драйверами и сервисами.

## Структура дерева (ключевые каталоги)

- `boot` — ранняя загрузка и подготовка к long mode.
- `kernel/core` — arch-независимые интерфейсы.
- `kernel/common` — общие подсистемы ядра.
- `kernel/arch/x86_64` — текущая рабочая реализация.
- `kernel/fabric` — Fabric: bus/device/driver/service.
- `kernel/input` — InputCore.
- `kernel/interrupts` — общие части подсистемы прерываний.
- `drivers` — Fabric-драйверы.
- `include` — публичные заголовки ядра.

## Разделение ответственности

- Arch-независимые интерфейсы задают контракты и типов/структур данных.
- Arch-специфичные реализации предоставляют реальное поведение.
- Fabric предоставляет единый путь обнаружения и подключения устройств.
- Файловые системы, сетевой стек и драйверы — внутри ядра.
- Минимизация переходов user↔kernel — стратегический выбор.
- Архитектура модульная, подсистемы слабо связаны логически.
- Применяется изоляция и проверки, нехарактерные для классических монолитов.

## Изоляция и проверки (что это означает)

- Явные контракты подсистем: входы, выходы, владение ресурсами.
- Валидация границ и параметров на стыках подсистем.
- Минимизация общих глобальных структур и скрытых зависимостей.
- Проверки корректности состояния перед переходами фаз инициализации.
- Регистрация и обнаружение устройств через Fabric, а не прямые ссылки.
- Сервисы публикуются и потребляются по типам, а не по “знакомству”.
- Предпочтение статически проверяемых API вместо неявных соглашений.
- Диагностика должна выявлять нарушение контракта, а не маскировать его.

## IDL и генерация кода (границы слоев)

Для жесткой дисциплины границ вводится IDL и генератор кода по модели
Apple MIG: интерфейс описывается в `.defs`, после чего генератор создает
клиентские и серверные стабы, а также диспетчеры. Это снижает ручное
маршаллирование и делает границы между слоями проверяемыми.

## Контракты на уровне подсистем

- Подсистемы должны публиковать явные интерфейсы в `include`.
- Взаимодействие через Fabric предпочтительнее прямых связей.
- Все публичные API фиксируются в документации подсистемы.

## Linker‑символы (инварианты памяти)

Эти символы считаются обязательными для корректного управления памятью ядра:

- `__kernel_start`, `__kernel_end`.
- `__text_start`, `__text_end`.
- `__rodata_start`, `__rodata_end`.
- `__data_start`, `__data_end`.
- `__bss_start`, `__bss_end`.
- `__init_start`, `__init_end` (память init‑секций освобождается после старта).

## Текущее состояние (кратко)

- x86_64 — рабочий основной таргет.
- Настроены GDT/IDT, ISR/IRQ и базовые таймеры.
- Есть PMM и заготовки под VMM.
- Fabric зарегистрирован и используется для HID клавиатуры.

## Планы

- Устойчивый VMM и выделение ядра в higher-half.
- VM-архитектура по слоям `vm_map -> vm_object -> pmap` с fault/COW/pager path.
- Минимальный планировщик и потоки.
- VFS и RAMFS.
- Системные вызовы и минимальный userspace.
- SMP: глобальная очередь на v0, затем per‑CPU run‑queues и балансировка.
