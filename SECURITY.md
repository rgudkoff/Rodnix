# Security Policy

## Scope

This document defines how vulnerabilities should be reported and what the
project expects during handling and disclosure.

## Reporting Vulnerabilities

Public issues and public pull requests must not be used for initial disclosure
of security vulnerabilities.

Reports should be sent through the private security or maintainer channel used
by your team or by the installation that hosts this repository.

If a dedicated security channel has not yet been established, use the following
fallback process:

1. share the issue only with repository maintainers;
2. do not publish PoC details, exact exploitation steps, or operational data;
3. wait for acknowledgement and coordinated next steps.

## What to Include

- a short description of the issue;
- the affected component or repository path;
- reproduction conditions;
- expected impact;
- the version or commit where the issue reproduces;
- when available, a minimal PoC and a workaround.

## Handling Expectations

The maintaining side should:

- acknowledge receipt;
- perform an initial impact assessment;
- define a fix or mitigation plan;
- agree on the disclosure form and timing.

## Disclosure

Public disclosure is acceptable only after:

- a fix is available;
- a workaround is documented, if needed;
- maintainers and the reporter agree on the disclosure window.

## Supported Versions

The supported baseline is the active development branch and the latest internal
stable baseline, if such a baseline is explicitly designated by the team.

If release/version policy becomes more formal, this section should be updated
together with the release process.
